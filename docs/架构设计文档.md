## 1. 架构概述

### 1.1 设计哲学

lite-interpreter 的核心设计哲学是**受控的智能**——不是让 LLM 自由发挥的通用 Agent，而是一个将数据分析任务编排为可治理、可观测、可回放、可交付的工程化系统。

四个核心设计原则：

| 原则 | 含义 | 实现机制 |
|------|------|----------|
| **可治理** | 所有动态行为必须经过策略审批 | HarnessGovernor + AST 审计 |
| **可观测** | 每一步状态变化都有迹可循 | Event Bus + Event Journal |
| **可回放** | 任务中断后可从断点恢复 | Blackboard checkpoint + 节点级持久化 |
| **可交付** | 结果可下载、可引用、可追溯 | 结构化产物 + 审计链 |

### 1.2 架构总览

系统采用**六层架构**，自上而下职责递减、控制力递增：

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 产品层  apps/web (React/Vite)              │
├─────────────────────────────────────────────────────┤
│  Layer 2: 产品API层  /api/app/* (Starlette/Pydantic) │
├─────────────────────────────────────────────────────┤
│  Layer 3: 控制层  Blackboard + Event Bus + Journal    │
├─────────────────────────────────────────────────────┤
│  Layer 4: 编排层  DAG Engine (静态链 + 动态链)        │
├─────────────────────────────────────────────────────┤
│  Layer 5: 动态运行层  DeerFlow Sidecar               │
├─────────────────────────────────────────────────────┤
│  Layer 6: 执行与知识层  Sandbox + KAG + SkillNet     │
└─────────────────────────────────────────────────────┘
         ↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓
┌─────────────────────────────────────────────────────┐
│  Storage: Postgres + Qdrant + Neo4j                  │
└─────────────────────────────────────────────────────┘
```

> 详细架构图见 `docs/architecture-diagram.png`

## 2. 各层架构详解

### 2.1 Layer 1 — 产品层

**技术栈**：React 18 + TypeScript + Vite + React Router + TanStack Query + Tailwind CSS

**设计决策**：从 Streamlit 快速原型迁移到真实 Web 前端，是产品面工程化的关键一步。

**页面结构**：

| 路径 | 功能 |
|------|------|
| `/analyses` | 分析总览，展示最近分析状态与快速入口 |
| `/analyses/new` | 新建分析，选择资料与提问 |
| `/analyses/:id` | 分析详情，业务视图 / 运行时视图切换 |
| `/assets` | 资料库管理 |
| `/methods` | SkillNet 沉淀方法查看 |
| `/audit` | 治理与审计记录 |

**前端只消费 `/api/app/*` 合同**，所有展示数据来自真实 API 响应，不伪造内部状态。这是防止"两套真相"的关键约束。

### 2.2 Layer 2 — 产品 API 层

**入口**：`src/api/main.py`（Starlette ASGI 应用）

**核心模块**：

| 模块 | 职责 |
|------|------|
| `app_router.py` | 定义 `/api/app/*` 路由 |
| `app_schemas.py` | Pydantic 请求/响应模型，API 合同的唯一真相源 |
| `app_presenters.py` | 将 Blackboard 状态投影为前端可消费的 DTO |
| `auth.py` | Bearer Token 认证 + AuthContext 构建 |
| `task_flow_service.py` | 任务流门面：创建分析 → 启动 DAG → 投影结果 |
| `asset_service.py` | 资料上传与管理 |

**API 合同**：

```
GET  /api/app/session                     → 会话 bootstrap
GET  /api/app/analyses                    → 分析列表（分页）
POST /api/app/analyses                    → 创建分析
GET  /api/app/analyses/{id}               → 分析详情
GET  /api/app/analyses/{id}/events        → 事件流（SSE）
GET  /api/app/analyses/{id}/outputs/{oid} → 产物下载
GET  /api/app/assets                      → 资料列表
POST /api/app/assets                      → 上传资料
GET  /api/app/methods                     → 方法列表
GET  /api/app/audit                       → 审计记录
```

**认证模型**：Bearer Token → `AuthContext`（token, subject, tenant_id, workspace_id, role, grants）。角色层级：viewer(10) < operator(20) < admin(30)。

### 2.3 Layer 3 — 控制层

控制层是系统的**状态事实源**，所有模块通过 Blackboard 读写状态，通过 Event Bus 传播变更。

#### Blackboard 体系

| 类 | 职责 |
|----|------|
| `GlobalBlackboard` | 全局单例，管理任务生命周期，子黑板注册中心 |
| `ExecutionBlackboard` | 单任务执行状态：DAG 节点 checkpoint、动态链状态 |
| `KnowledgeBlackboard` | 知识状态：检索计划、证据包、上下文快照 |
| `MemoryBlackboard` | 任务级记忆：历史交互、偏好、累积上下文 |

**GlobalBlackboard 核心特性**：

- 线程安全：`threading.RLock` 保护所有状态读写
- 持久化优先：每次状态变更后写入 Postgres
- 幂等创建：`idempotency_key` + `request_fingerprint` 防止重复创建
- 状态机：`PENDING → ROUTING → ANALYZING → CODING → EXECUTING → SUCCESS/FAILED → ARCHIVED`

#### Event Bus & Journal

- `AsyncEventBus`：全局单例，后台独立线程运行 asyncio 事件循环，发布-订阅模式
- `EventJournal`：事件持久化追加日志，支持审计回放
- 隐私保护：发布前通过 `mask_payload` 自动脱敏 PII

### 2.4 Layer 4 — 编排层：DAG Engine

DAG Engine 是系统的**主流程 owner**，负责将任务编排为有序的节点执行链。

#### 静态链（Static Flow）

```
Router → DataInspector → KAGRetriever → ContextBuilder → Analyst
       → StaticEvidence → EvidenceCompiler → Coder → Auditor
       → [Debugger → Auditor] → Executor → [Debugger → Auditor]
       → SkillHarvester → Summarizer
```

**关键节点职责**：

| 节点 | 职责 |
|------|------|
| `RouterNode` | 根据复杂度评分决定走静态链还是动态链 |
| `DataInspector` | 结构化数据探查：读取 CSV/TSV/JSON，生成数据概况 |
| `KAGRetriever` | 调用 KAG QueryEngine 获取相关知识证据 |
| `AnalystNode` | 基于数据+知识生成可执行分析规划 |
| `CoderNode` | 根据规划生成 Python 代码 |
| `AuditorNode` | AST 静态分析，检查安全与质量 |
| `ExecutorNode` | Docker 沙箱中执行代码 |
| `SummarizerNode` | 生成最终分析报告 |
| `SkillHarvesterNode` | 从执行结果中提取可复用技能 |

#### 动态链（Dynamic Flow）

当 Router 判定任务复杂度超过阈值时进入：

```
Router → DynamicSwarm → [DeerFlow Sidecar] → 回流静态链
```

**设计约束**：DAG 仍是 owner，DeerFlow 只是受控的动态研究 sidecar，不拥有最终 Python 执行边界。

#### Checkpoint 机制

每个节点执行前后通过 `_run_checkpointed_node()` 写入 `ExecutionBlackboard`：

```python
# 执行前：标记 running
checkpoints[node_name] = {"status": "running", "started_at": ..., "attempt_count": N+1}

# 执行后：标记 completed + 存储 output_patch
checkpoints[node_name] = {"status": "completed", "completed_at": ..., "output_patch": ...}

# 异常时：标记 failed + 记录 error
checkpoints[node_name] = {"status": "failed", "failed_at": ..., "error": str(exc)}
```

支持任务中断后从断点恢复，也支持跳过已完成节点。

### 2.5 Layer 5 — 动态运行层：DeerFlow Sidecar

| 组件 | 职责 |
|------|------|
| `DynamicSupervisor` | 构建 TaskEnvelope、评估治理策略、准备 DeerFlow 请求 |
| `DeerflowBridge` | DeerFlow 通信桥：发送任务请求、接收研究轨迹 |
| `TraceNormalizer` | 将 DeerFlow 返回的研究轨迹转换为标准格式 |
| `DynamicContextEnvelope` | 构建注入 DeerFlow 的权威上下文 |

**治理流程**：`DynamicSupervisor.prepare()` → `HarnessGovernor.evaluate_dynamic_request()` → 通过/拒绝。被拒绝时返回 `denied_patch`，不静默回退。

### 2.6 Layer 6 — 执行与知识层

#### Sandbox：安全执行环境

采用 **AST 审计 + Docker 沙箱**双重隔离：

1. **AST 审计**（`ast_auditor.py`）：解析代码 AST，检查高风险模块/内建函数/方法调用
2. **Docker 执行**（`docker_executor.py`）：创建容器、挂载数据、执行代码、收集结果
3. **安全策略**（`security_policy.py`）：高风险模块列表、容器资源限制

#### KAG：知识增强生成

四个子模块构成完整的知识闭环：

```
Builder (知识构建)         Retriever (知识检索)
  Chunker                    QueryEngine
  EntityExtractor            HybridSearch (BM25 + 向量 + 图谱)
  RelationExtractor          Rerank (Cross-encoder)
  Embedding                  Dedup (语义去重)
  Fusion
                           Context (上下文管理)
Compiler (知识编译)          Compressor / Selector / Formatter
  KnowledgeCompilerService
  KnowledgeSpecParser
```

**检索策略**：Hybrid Search 融合 BM25（关键词精确匹配）+ Qdrant 向量检索（语义匹配）+ Neo4j 图谱检索（关系遍历），通过 RRF 排序融合结果。

#### SkillNet：技能沉淀网络

| 组件 | 职责 |
|------|------|
| `SkillHarvester` | 从执行轨迹中提取可复用技能候选 |
| `SkillValidator` | 验证技能的有效性和安全性 |
| `SkillPromoter` | 将验证通过的技能提升为预设技能 |
| `SkillRetriever` | 根据查询匹配已有技能 |

预设技能类别：可视化、统计分析、数据清洗、合规检查。

## 3. 核心数据流

### 3.1 主链路：分析创建 → 执行 → 结果

```
[前端] POST /api/app/analyses
  │
  ▼
[TaskFlowService.create_analysis()]
  │ → GlobalBlackboard.create_task()
  │ → 发布 UI_TASK_CREATED 事件
  │ → 持久化到 Postgres
  │
  ▼
[TaskFlowService.start_analysis()]
  │ → 提交到 TASK_FLOW_EXECUTOR 线程池
  │ → dag_graph.run(task_id, nodes)
  │
  ▼
[DAG Engine 静态链]
  │ Router → DataInspector → KAGRetriever → ContextBuilder
  │ → Analyst → Coder → Auditor → Executor → Summarizer
  │ 每个节点执行后更新 ExecutionBlackboard checkpoint
  │ 每次状态变更通过 EventBus 发布事件
  │
  ▼
[EventBus 分发事件]
  │ → EventJournal 持久化
  │ → SSE 推送给前端（实时进度）
  │
  ▼
[前端] GET /api/app/analyses/{id}
  │ AppPresenters 将 Blackboard 状态投影为 DTO
  │ 返回：status, progress, outputs, warnings
```

### 3.2 动态研究回流

```
[Router] complexity_score > threshold
  │
  ▼
[DynamicSwarmNode]
  │ → DynamicSupervisor.prepare()
  │ → HarnessGovernor 审批
  │ → 构建 DeerflowTaskRequest
  │
  ▼
[DeerFlow Sidecar] 执行动态研究
  │ → 返回研究轨迹 + resume_overlay
  │
  ▼
[TraceNormalizer] 归一化轨迹
  │
  ▼
[DAG Engine] 回流静态链继续执行
```

### 3.3 知识构建与检索

```
资料上传 → Chunker 分块 → Postgres (全文) + Qdrant (向量)
                        → EntityExtractor + RelationExtractor → Neo4j (图谱)

查询时：QueryEngine → HybridSearch(BM25 + Qdrant + Neo4j) → Rerank → Context
```

## 4. 存储架构

三存储引擎分离，各司其职：

| 存储引擎 | 引擎 | 数据类型 | 客户端 |
|----------|------|----------|--------|
| 状态存储 | Postgres | Blackboard 状态、审计记录、任务元数据 | `PostgresDBClient` → `StateRepo` / `AuditRepo` |
| 向量存储 | Qdrant | 文档分块向量表示（cosine/dot/L2） | `VectorClient` |
| 图谱存储 | Neo4j | 知识图谱（实体+关系，semantic/temporal/causal/entity） | `GraphClient` |

**设计权衡**：每个引擎做最擅长的事，可独立扩展；代价是运维复杂度增加，跨存储事务一致性需应用层保证。

## 5. 安全架构

### 5.1 认证与授权

- Bearer Token 认证，Token 配置通过 `API_AUTH_TOKENS_JSON` 注入
- `AuthContext` 携带 tenant_id + workspace_id + role + grants
- 所有数据操作按 `(tenant_id, workspace_id)` 隔离

### 5.2 代码执行安全

双重隔离机制：

1. **AST 审计**（第一层）：解析代码 AST，拦截高风险模块导入（os、subprocess、socket 等）、高风险内建函数（eval、exec、__import__ 等）、高风险方法调用（__subclasses__、__globals__ 等）
2. **Docker 沙箱**（第二层）：容器内执行，限制 CPU/内存/时间资源，产物通过卷挂载收集

### 5.3 数据安全

- 路径安全：产物下载限制在当前任务的 tenant/workspace 目录内
- PII 保护：Event Bus 发布前自动脱敏
- 多租户隔离：Blackboard 按 `(tenant_id, task_id)` 隔离状态

## 6. 容错与可靠性

### 6.1 分层异常体系

```
SandboxBaseError → AuditFailError / CodeExecError / DockerOperationError / ExecTimeoutError
BlackboardBaseError → TaskNotExistError / SubBoardNotRegistered / StatusUpdateError
DAGBaseError → TaskLeaseLostError
```

### 6.2 容错机制

| 机制 | 说明 |
|------|------|
| 节点级重试 | Coder → Debugger → Auditor 循环，最多 MAX_RETRIES（默认 3）次 |
| 任务租约 | TTL 60s + 心跳 20s，防止多进程同时执行同一任务 |
| 中断恢复 | 服务重启后通过 `list_unfinished_tasks()` 恢复，节点 checkpoint 支持断点续执行 |
| 治理拒绝 | HarnessGovernor 可拒绝动态运行请求，返回 `denied_patch` |

### 6.3 降级策略

关键服务不可用时明确返回 `unavailable` 语义，不静默回退。DeerFlow 不可用时不悄悄降级为静态链。

## 7. 关键设计决策与权衡

### 7.1 Blackboard 作为单一状态源

**决策**：所有模块通过 Blackboard 读写状态，不允许隐式状态传递。

**权衡**：
- ✅ 状态可追溯、可持久化、可回放，模块间解耦
- ❌ 增加一次间接层，需要严格的线程安全设计

### 7.2 DAG 为主、DeerFlow 为辅

**决策**：DAG Engine 始终是流程 owner，DeerFlow 只作为受控 sidecar。

**权衡**：
- ✅ 主流程确定性强，可预测、可审计
- ❌ 动态研究能力受限于预设的 step 和 token 预算

### 7.3 AST 审计 + Docker 沙箱双重隔离

**决策**：代码执行前先做 AST 静态审计，通过后在 Docker 容器中执行。

**权衡**：
- ✅ 双层防护，AST 拦截明显恶意代码，Docker 隔离运行时风险
- ❌ AST 审计可能有误报，Docker 容器启动有额外延迟

### 7.4 三存储引擎分离

**决策**：Postgres 存状态、Qdrant 存向量、Neo4j 存图谱。

**权衡**：
- ✅ 每个引擎做最擅长的事，可独立扩展
- ❌ 运维复杂度增加，跨存储事务一致性需应用层保证

### 7.5 节点级 Checkpoint

**决策**：每个 DAG 节点独立 checkpoint，而非全局快照。

**权衡**：
- ✅ 粒度细，可跳过已完成节点恢复，存储开销小
- ❌ 节点间状态依赖需通过 `output_patch` 显式传递

## 8. 技术栈总结

| 层级 | 技术 |
