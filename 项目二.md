# Lite-Interpreter 项目综述（2026-04-23）

## 1. 这个项目现在到底是什么

`lite-interpreter` 现在是一个**面向财务、会计与经营分析场景的受控分析运行时原型**。

当前成熟度、测试基线与已知热点统一以 `docs/reference/project-status.md` 为准；本文件侧重中文综述与工程判断。

它不是：

- 一个让 agent 自由发挥的通用自治系统
- 一个只会“写一段 Python demo”就结束的分析玩具
- 一个继续拿 Streamlit 包装成正式产品面的实验壳子

它是：

- 用确定性 DAG 托住主链路
- 用 DeerFlow sidecar 处理复杂动态研究
- 用 Blackboard 管理任务状态、执行状态、知识状态和记忆状态
- 用 sandbox 保住最终代码执行边界
- 用真实 Web 前端和 `/api/app/*` 合同把结果稳定交付出去

## 2. 这轮大改到底完成了什么

### 2.1 产品面从“临时界面”变成“真实前端”

这次最重要的工程事实，不是换了个配色，也不是多了几个页面，而是：

1. 前端已经硬切到 `apps/web` 下的 React/Vite 工作台
2. 会话、工作区、分析、资料、方法、审计已经形成统一工作台
3. 前端消费的是 app-facing 合同，而不是后端内部读模型碎片

这意味着项目终于不再像“给开发者调试用的控制台界面”，而开始像一个真的数据分析平台骨架。

### 2.2 旧产品接口已经收口

现在真正的产品面接口已经统一到：

- `/api/app/session`
- `/api/app/analyses`
- `/api/app/assets`
- `/api/app/methods`
- `/api/app/audit`

这一步很重要，因为它把“前端看到的世界”和“后端内部执行细节”切开了。

旧 `tasks/executions/uploads/...` 风格的公开产品接口不该再回来；那些更适合内部读模型、调试或历史兼容，不适合继续当产品合同。

### 2.3 配置面开始收口

以前这类项目最容易烂掉的地方之一，就是配置散得到处都是，最后没人知道哪个才是真。现在这方面已经开始走向可维护：

- `config/` 放仓库可跟踪的默认策略
- `.env` 放本地运行环境和密钥
- `.env.example` 作为配置契约模板
- 文档开始围绕这个配置面重写，而不是继续靠记忆和口口相传

## 3. 当前架构判断

### 3.1 做对了的地方

1. **owner 边界清楚**
   - DAG 仍是系统 owner
   - sandbox 仍是最终代码执行 owner
   - DeerFlow 只是动态研究 runtime，不是系统 owner

2. **控制面不是日志拼接，而是事实源**
   - `TaskEnvelope`
   - `ExecutionIntent`
   - `DecisionRecord`
   - `ExecutionRecord`
   已经形成跨模块共享契约

3. **产品面终于接上了工程闭环**
   - 资料上传有稳定 `assetId`
   - 创建分析通过 `assetIds` 显式挂接输入
   - 详情页能看结论、事件和结果产物
   - 会话与工作区切换不再是临时拼接

4. **文档与项目结构开始变得可读**
   - 主文档职责拆清了
   - `directory.txt` 回到目录导览本职
   - `docs/reference/project-status.md` 成为状态真相源

### 3.2 仍然最值得盯的地方

1. **控制面仍然复杂**
   状态模型已经不少，后续真正需要做的是：
   - 继续减少跨域互踩
   - 保持 section 级持久化边界
   - 让冷路径错误稳定、不炸 API

2. **静态 codegen 还偏模板化**
   `analyst -> coder -> static_codegen` 这条链已经可用，但 `analysis_plan`、skill hints 和知识上下文对最终执行策略的影响还不够深。

3. **前端虽然已经成型，但交互还没到“成熟产品”**
   现在已经不像控制台了，但还可以继续做：
   - 长任务进度反馈更细粒度
   - 结果产物的预览与下载更顺手
   - 方法与审计页更少“辅助页感”

4. **文档治理不能反弹**
   这轮好不容易把结构理顺，如果以后每次改前端/API 都不改文档，很快又会回到“每份文档都在讲一部分旧真相”的状态。

## 4. 当前用户主链路

一个任务现在合理的主链路是：

1. 用户在 Web 工作台进入当前 workspace
2. 上传结构化数据或业务文档到资料库
3. 新建分析，并显式勾选本次要引用的资料
4. Router 判断走静态链还是动态链
5. 若走静态链：
   - Data Inspector
   - KAG Retriever
   - Context Builder
   - Analyst
   - Coder
   - Auditor
   - Executor
   - Skill Harvester
   - Summarizer
6. 若走动态链：
   - Dynamic Supervisor
   - DeerFlow Bridge
   - Dynamic trace 回写
   - 需要时回流静态链
7. 前端读取 app-facing 读模型，展示结论、证据、事件和产物

## 5. 当前版本最重要的工程判断

### 5.1 安全和可控优先于“看起来更智能”

这轮收口之后，项目明确拒绝了几类坏方向：

- 不把最终代码执行权交给动态 runtime
- 不重新开放 query-string token
- 不再把任意绝对路径当成前端可读产物
- 不恢复 embedded / auto runtime mode

### 5.2 一致性优先于“再堆几层功能”

这轮真正提升质量的，不是功能数量，而是系统边界：

- 产品前端有了稳定合同
- 输入、执行、产物、审计走同一条链
- 文档、配置、路由开始往一套真相收敛

### 5.3 文档要服务协作，不要制造第二套系统

好的仓库文档不是越多越好，而是每份文档只负责一件事：

- README 负责入口
- project status 负责状态真相
- architecture 负责解释边界
- deployment 负责运行
- development guide 负责改动流程
- testing 负责验证方法

## 6. 下一阶段最值得做什么

如果继续推进，我建议优先顺序是：

1. **继续强化结果产物体验**
   现在已经有受控内容 API，下一步该做的是更顺手的预览、下载和结果摘要体验。

2. **补强长任务交互反馈**
   真正给财务/经营分析团队用的系统，不能只在任务跑的时候让用户盯着一串模糊状态。

3. **继续推进 skill usage/outcome 原子计数**
   让方法沉淀的数据在并发下也可靠。

4. **持续压缩文档和配置漂移**
   每次改动前端、API、配置面，都应该同步更新对应文档，不要再让仓库出现历史残影。

## 7. 结论

这个项目现在最有价值的地方，不是它有多少“AI 能力名词”，而是它终于开始像一个可维护、可协作、可交付的分析平台骨架了。

接下来真正该做的，不是回头继续堆旧接口和临时页面，而是把这条主链做得更稳：

> 路由正确，状态一致，研究可控，执行安全，结果可交付，方法能沉淀。

把这条链做稳，项目就站住了。
