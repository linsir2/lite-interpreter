# Lite-Interpreter 项目综述（2026-04-20）

## 1. 这个项目现在到底是什么

`lite-interpreter` 现在是一个**面向数据分析任务的受控执行运行时原型**。

当前成熟度、测试基线与已知热点统一以 `docs/project_status.md` 为准；本文件侧重中文综述和工程判断。

它不是：

- 一个让 agent 自由发挥的通用自治系统
- 一个只会“写一段 Python demo”就结束的分析玩具
- 一个把所有能力都塞进单个模型 prompt 的黑箱

它是：

- 用确定性 DAG 托住主链路
- 用 DeerFlow sidecar 处理复杂动态研究
- 用 Blackboard 管理任务状态、执行状态、知识状态和记忆状态
- 用 sandbox 保住最终代码执行边界
- 用 KAG 和 SkillNet 把知识与经验沉淀成可复用资产

## 2. 当前架构判断

### 2.1 做对了的地方

1. **owner 边界清楚**
   - DAG 仍是系统 owner
   - sandbox 仍是最终代码执行 owner
   - DeerFlow 只是动态研究 runtime，不是系统 owner

2. **控制面不再只是日志拼接**
   - `TaskEnvelope`
   - `ExecutionIntent`
   - `DecisionRecord`
   - `ExecutionRecord`
   已经形成跨模块共享契约

3. **动态链和静态链已经开始真正融合**
   - 可以先动态研究，再回流静态执行
   - 成功路径可以沉淀 skill
   - 前端已经能消费 execution/tool-call/workspace 读模型

4. **工程化开始收口**
   - 统一测试命令
   - runtime capability manifest
   - diagnostics / conformance
   - execution resources

### 2.2 仍然最值得盯的地方

1. **控制面仍然复杂**
   现在状态模型已经不少，真正需要持续做的是：
   - 继续减少跨域互踩
   - 保持 section 级持久化边界
   - 让冷路径错误稳定、不炸 API

2. **静态 codegen 还偏模板化**
   目前 `analyst -> coder -> static_codegen` 这条链已经可用，但很多“计划”和“skill hint”仍更多体现在 payload 和输出解释里，真正改变执行逻辑的程度还不够。

3. **前端主工作台还在收口期**
   现在已经比之前统一很多，但还需要继续做：
   - task input 可见性
   - 更清晰的 artifact 消费 API
   - 更稳定的轮询/流式体验

4. **格式支持要诚实**
   这版明确把结构化静态执行的可靠格式收敛到 `csv/tsv/json`。这不是退步，是把“假支持”改成“真边界”。

## 3. 当前用户主链路

一个任务现在的合理主链路是：

1. 用户在 Analysis Workspace 创建任务
2. 上传结构化数据或业务文档
3. Router 判断走静态链还是动态链
4. 若走静态链：
   - Data Inspector
   - KAG Retriever
   - Context Builder
   - Analyst
   - Coder
   - Auditor
   - Executor
   - Skill Harvester
   - Summarizer
5. 若走动态链：
   - Dynamic Supervisor
   - DeerFlow Bridge
   - Dynamic trace 回写
   - 需要时回流静态链
6. 前端读取 server-built workspace payload 和 execution resources

## 4. 当前版本最重要的工程判断

### 4.1 安全优先于“看起来更智能”

这轮稳定化之后，项目明确收紧了几条边界：

- `tenant_id/workspace_id` 不能再随便进路径和 DB 标识符
- API 不再默认裸奔
- session secret 不能再用固定默认值
- artifact path 不再把任意绝对路径直接交给前端
- debugger fallback 不再制造“成功假象”

### 4.2 一致性优先于“多写几个模块”

这轮比起加新功能，更重要的是修：

- section 级状态持久化
- dynamic_then_static 摘要一致性
- KAG 假 parsed 问题
- durable memory 不可用时的降级语义

### 4.3 文档要服务工程，不要制造噪音

仓库之前的 markdown 文档过多，而且彼此重叠、漂移、叙事断裂。现在应该把文档压缩成少数几份真文件：

- 总览
- 架构
- 开发
- 部署
- 测试
- 状态真相源

## 5. 下一阶段最值得做什么

如果下一阶段继续推进，我建议优先顺序是：

1. **artifact 内容读取 API 化**
   让前端不再只显示路径和禁用提示，而是可以通过受控接口读取/下载产物。

2. **workspace 资产显式挂接 task**
   让“先上传，再分析”真正变成稳定产品流，而不是现在依赖 task 内上传。

3. **memory usage/outcome 原子计数**
   让 skill usage telemetry 在并发下不丢数。

4. **静态 codegen 从模板进一步走向策略化**
   真正让 `analysis_plan`、`approved_skills`、`compiled_knowledge` 改变生成代码，而不是只改解释文案。

## 6. 结论

这个项目现在已经有一个清晰方向：

- 不是继续堆概念
- 不是继续加更多 runtime 模式
- 不是继续加更多“看起来厉害”的页面

而是把现有这条主链做扎实：

> 路由正确，状态一致，知识可追踪，动态可控，执行安全，结果可解释，经验能沉淀。

把这条链做稳，项目就有价值。再往上加东西，才不会变成噪音。
